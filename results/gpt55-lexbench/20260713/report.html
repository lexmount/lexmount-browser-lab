GPT-5.5 × LexBench：Lexmount 稳定性与并发复核

这份报告把全量配对、5090 失败任务复测和 c10/c64 容量实验放在同一证据框架中。 核心判断是：Lexmount 在本实验里有清晰的端到端优势，但优势主要属于完整 browser backend， 不能被解释为 Chromium 内核本身更强。

结论 全量配对 机制审计 5090 复测 c64 容量 资源成本 方法与来源

Decision

可以下的结论，比“远端浏览器更强”更具体

证据支持

在固定模型、任务集、Agent 版本和 Judge 的 210 条配对实验中， Lexmount 完整 browser backend 的成功率高于 Mac 本地 Chrome 13.33 个百分点。 对原始 Local failure 做偏向 Local 的 5090 敏感性分析后，优势仍为 8.57 个百分点。

证据不支持

不能说远端 Chromium 内核“更聪明”，也不能说持续 64 并发已经跑满。 当前实验同时改变了出口、地区、指纹、会话运行时与宿主资源。

Resource and cost

Lexmount 降低 runner 压力，但没有证明总资源更低

原始全量实验中，Lexmount 列只采集本机 runner 控制进程，不包含远端浏览器宿主机； Local 列包含 runner、Agent 与本地 Chrome 进程树。两列适合回答“本机压力是否被移出”，不适合比较服务端总成本。

Decision boundary

下一步该验证什么

现在可以用于产品结论

“在 GPT-5.5 × LexBench 的固定配置中，Lexmount 完整 browser backend 提供了更好的站点可达性与端到端结果； 优势在偏向 Local 的 5090 failure-only 敏感性分析后仍存在。”

现在不能用于产品结论

“Lexmount 的 Chromium 内核更强”“所有站点都更稳定”“c64 已持续跑满”“Lexmount 服务端总资源低于 Local”。

最小下一轮实验

让两端共享同一出口，记录 UA、browser build、IP/ASN/geo；全量至少两轮交叉顺序； 对 discordant 与阈值附近任务重复 Judge；c64 采用预创建并要求稳定 active=64。

主要限制

  • 原始 210 条总体结果仍是单次顺序运行；12 条机制复测按机制选样，不能替代全量多轮。
  • Agent 没有固定 seed；Judge temperature 为 1.0 且每条只判一次。
  • 5090 只重跑原始 Local failure，132/210 是偏向 Local 的敏感性分析，不是独立总体估计。
  • c10 session sampler 的完整曲线缺失，只保留运行中 API 快照；c64 sampler 有 144 个样本且无采样错误。
  • 资源指标跨 macOS RSS、Linux PSS/cgroup 与远端浏览器宿主边界，必须按表中口径阅读。
