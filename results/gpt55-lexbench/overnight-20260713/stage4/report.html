决策问题

Lexmount 为什么看起来更强，真实优势到底在哪里？

把总体准确率、站点差异、runner 资源和 session admission 分开回答，不用一个总分覆盖四种机制。

短答案

不能说 Lexmount 通用更准；可以说它在部分站点更稳，并把 Chrome 负载移出了 runner。

Stage 1 的反序重复仅为 +1.56pp，区间跨越双方优势；Stage 2 证明站点方向大多可复现，但 Lexmount-only 与 Local-only 同时存在；Stage 3 的 c32 又回到 40:41。真正稳定的产品收益是部分访问敏感站点的成功方向、干净的 session 生命周期，以及明显更低的 runner CPU/内存。运行默认先用 c32；c64 等任务型 admission gate 通过再测。

01 / evidence ladder

三个阶段回答三个不同问题，不能混成一个新总分

总体样本负责判断通用赢家；选择后的 holdout 负责检验机制方向；容量矩阵负责判断可运行并发与资源。每层只回答自己的问题。

02 / claim boundary

可以对外说什么，必须停在哪里

产品表述只保留能由当前证据直接支持的部分；因果、全量、长期和端到端成本都明确留在边界外。

证据允许

  • 本轮没有建立 Lexmount 或 Local 的通用准确率赢家。
  • 一组访问敏感站点上，Lexmount 的成功方向可重复。
  • 另一组控制链或证据敏感任务上，Local 的成功方向也可重复。
  • Lexmount 将本地 Chrome 负载移出 runner；c32 runner CPU/PSS 明显更低。
  • 账号实际可同时持有 64 个 session，但任务型 28/36 高水位分配仍不可靠。

证据禁止

  • “Lexmount 在 LexBench 上普遍更准”。
  • “优势已经证明来自 IP、代理或浏览器指纹”。
  • “配额 64 就等于任意 profile mix 稳定 64 并发”。
  • “controller PSS 就是远端浏览器端到端总资源”。
  • “c16 到 c32 的一轮变化证明并发导致质量变化”。
建议采用的产品表述

在本轮 GPT-5.5 + LexBench 固定样本中，Lexmount 与 Local Chrome 的总体任务质量接近；Lexmount 在部分访问敏感站点上提供可重复的稳定性收益，并显著降低 runner 侧浏览器资源。Local 在部分当前控制链敏感任务上更稳，因此后端路由与 fallback 比单一总分更有价值。

把 c32 作为有完整质量、资源、真实 session 峰值与清理证据的默认并发。

每次启动

使用实际 EN/ZH 分配做 raw admission preflight；正式 arm 必须观察到命名并发、monitor error 0、residual 0。

开放 c64

先连续两次通过 28 EN + 36 ZH，或取得 provider reservation/profile allocation 证据，再运行正式 c64。

故障回退

把稳定站点方向当候选 routing/fallback，不把当前选择集直接写成线上规则。

05 / next decision-changing work

下一轮只做能改变决策的三件事

继续堆同一时间窗的一次性样本收益很低。优先打剩余两个不确定性：跨时段稳定性与高水位 admission。

Priority 1

再做至少两个跨时段 c32 反序 pair。 判断 40:41 的近似持平与 Local 1.214× 吞吐、2–4× runner 资源比是否持续。

Priority 2

复测任务型 28/36 admission。 连续 2/2 通过或拿到平台分配证据后，c64 才进入正式 Agent/Judge。

Priority 3

构造全新、未选择的站点路由 holdout。 检验访问敏感与控制敏感签名能否支持 backend-aware fallback，而不是复用已有赢家任务。

没有必要马上补一个名义 c64。

如果 preflight 没有实际达到 64，Agent 运行只能证明“配置写了 64”，不能回答 64 并发的质量或资源问题。先修 admission 可观测性，比多一行 c64 表格更能改变决策。

07 / self-audit

是否把三个阶段池化成新成功率？否总体样本、选择诊断集与并发处理保持分层。
