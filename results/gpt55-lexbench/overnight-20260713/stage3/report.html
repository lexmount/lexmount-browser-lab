Admission / concurrency / resources / boundary

阶段 3｜64 会话可达，正式可复现边界先定在 c32

把“账号配额是 64”拆成两个可证伪问题：平台能否同时给出 64 个活跃会话，以及同一批 64 条 LexBench 任务能否在 c16、c32 下完整运行并保持可控资源。

原始问题

并发开到 64 后，稳定性和资源会怎样？

先测原始 session admission，再测完整 Agent/Judge；达不到真实并发的运行不标成 c64。

短答案

64 配额真实可达，但今晚不能把它写成“默认负载稳定可用 64”。

32 EN + 32 ZH 的平衡探针真实观察到 64 个同时活跃会话，并连续保持 35 个采样点；同一任务集自然形成的 28 EN + 36 ZH 分配却两次均未达到 64。正式 c16/c32 四臂全部产出 64 轨迹和 64 Judge，最高可复现运行点因此先定为 c32。c32 质量为 40:41；Local 吞吐高 21.4%，runner CPU、PSS mean 与 kernel peak 则分别为 Lexmount 的 4.05×、2.09×、2.48×。

启动异常

第一次 c16 detached launch 因用户 systemd bus 环境缺失，在创建 scope 前失败；0 trajectory、0 session、residual 0，未进入正式矩阵。

根因

tmux shell 未继承 XDG_RUNTIME_DIR 与 DBUS_SESSION_BUS_ADDRESS，不是浏览器、平台容量或模型故障。

修复

launcher 根据当前 UID 恢复 user runtime 与 bus socket；替代 c16 和后续 c32 均从 detached shell 正常启动。

清理

每个 session 探针与正式 arm 结束后 residual session = 0；实验 cgroup 归零，本地既有无关 Chrome 进程未被终止。

05 / interpretation and next gate

下一步要解决的是 admission 可预期性，不是继续抬名义并发

阶段 3 把账号配额、瞬时 admission、完整 Agent 运行和 runner 资源四件事分开。只有同时满足，才有资格称为稳定并发档位。

证据支持

  • 账号至少能够同时持有 64 个活跃 Lexmount session。
  • c16 与 c32 下，Lexmount 和 Local 都完整产出 64 轨迹与 64 Judge。
  • Lexmount c32 实际 session 峰值达到 32，结束后无残留。
  • 同机 runner 口径下，Lexmount 的 CPU 与内存显著低于 Local；这是把 Chrome 移出 runner 的直接结果。

证据不支持

  • 默认 28 EN + 36 ZH 负载可稳定同时拿到 64 个 session。
  • 把一次 balanced 64 成功解释为任意 profile 组合均稳定。
  • 用 Lexmount controller PSS 代表端到端浏览器总资源。
  • 从各一轮 c16/c32 的成功率变化推断并发会提升或降低任务质量。
下一道正式 gate

先让 28 + 36 admission 在相同预检下连续成功至少 2 次，或提供平台侧 reservation / profile allocation 证据；再启动正式 c64。若只需今晚可复现的批处理，c32 是有完整轨迹、Judge、资源和生命周期证据的默认值。

06 / self-audit

阶段完成前的反问

每个“是”都要能回到机器产物；每个“否”都阻止报告越过当前证据边界。

反问回答依据 / 边界
64 quota 是否被直接观察，而非只读控制台？是32 + 32 探针 max = 64，target_sample_count = 35，结束 residual = 0。
自然任务 split 是否稳定拿到 64？否28 + 36 两次均失败，峰值为 49 与 46。
失败是否证明固定 49 或 ZH 25 上限？否25 + 25、32 + 32 与 ZH-only 36 均成功。
c32 是否真实观察到 32 个 Lexmount session？是正式 c32 monitor 的 active max 与 p95 均为 32。
四个正式 arm 是否均完整且无 synthetic Judge？是全部 64 / 64 / 64；synthetic failure remaining = 0。
启动失败是否污染正式样本？否失败发生在 scope 前，0 trajectory、0 session，并单独披露。
资源数字是否是端到端全系统成本？否Lexmount 只统计本地 controller；远端 browser host 不在 cgroup 内。
GPU 波动能否归因于浏览器 backend？否模型远端运行，主机存在无关 GPU 工作负载，GPU 仅作上下文。
一轮 c16/c32 能否给出质量因果关系？否它们可描述当前样本，但没有 concurrency replicate。
为何没有强行跑正式 c64？证据门槛未过任务同构 split 的 raw admission 0/2；强跑会把名义并发误写成实际并发。
