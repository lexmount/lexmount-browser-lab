Question / evidence / next test

阶段 1｜同机反序配对复测

把 GPT-5.5、Agent、Judge、64 条任务和 Linux 主机固定，只交换 Lexmount 与 Local Chrome 的运行顺序：之前看到的优势还能否复现？

原始问题

Lexmount 为什么看起来比本地 Chrome 强很多？

本阶段先排除机器差异、任务构成、执行顺序和残留资源，再讨论浏览器机制。

短答案

同机反序后，未复现“Lexmount 显著更强”。

两轮合计 Lexmount 82/128、Local 80/128，差值 +1.56 个百分点；按 64 个 task_id 聚类的 95% bootstrap CI 为 -9.38 至 +12.50 个百分点。可以确认 c10 运行完整、无 session 泄漏；不能据此宣布浏览器总体胜负。

05 / interpretation

把结论收紧到证据真正支持的范围

这一步不是否定 Lexmount，而是把产品稳定性、任务成功率和总基础设施成本拆开。

可以确认

  • 同机、同任务、同 c10 下，两端都能完整形成 64/64 轨迹与 Judge 结果。
  • Lexmount 两轮均无 session 创建失败、监控错误或残留 session。
  • Local rollout 更快，但 runner CPU、PSS 与 Chrome PSS 更高。
  • 存在 5 个连续 Lexmount-only 与 4 个连续 Local-only 的任务，值得做机制复测。

暂不能确认

  • Lexmount 在 LexBench 总体上显著优于 Local Chrome。
  • 上一轮 210-task 差值全部由浏览器后端造成。
  • runner 侧低 PSS 等于 Lexmount 端到端总资源更低。
  • 失败分类、GPU 值或单条轨迹足以证明具体根因。
当前最稳妥的产品表述

在 c10 下，Lexmount 提供了干净的远端 session 生命周期，并把浏览器内存与 CPU 从 runner 移出；Local Chrome 在这台机器上吞吐更高。两者的 Judge 成功率差在两轮 64-task 复测中不可区分。

06 / self-audit

阶段完成前的反问

每个“是”都对应可回溯产物；每个“否”都转成下一阶段边界，而不是被藏进脚注。

反问回答依据 / 边界
四个正式臂是否完整且同构？是每臂 64 planned / trajectory / judged；固定 task_id、Agent、模型、Judge、c10。
是否交换了 backend 顺序？是Pair A Local-first；Pair B Lexmount-first。
Judge 传输失败是否被记成浏览器 0 分？未发现四臂 synthetic Judge failure 均为 0；正常 artifact-only Judge 仍属有效观测。
Lexmount 是否有 session 泄漏或 quota 失败？否两轮 baseline 0、max 10、final 0、residual 0；session_create signal 0。
Local 是否触发资源守卫或留下本轮 Chrome？否guard 均为空；按启动时间和 cgroup 核对未发现本轮残留。未操作既有他人进程。
128 attempts 能否按 128 个独立样本估 CI？不能同一 task 的重复相关；CI 按 64 个 task_id cluster 重采样。
阶段 1 能否证明 Lexmount 总体更强？不能+1.56pp，95% CI [-9.38, +12.50]pp。
阶段 1 能否否定所有站点级优势？不能稳定单边成功仍有 9 个；需额外交叉复测。
runner 指标能否代表 Lexmount 总资源？不能远端 browser host 不在本地 cgroup。
GPU 能否解释差异？不能模型远端，且 GPU 有外部负载。
