Holdout replication / mechanism / boundary

阶段 2｜稳定差异存在，但不是单向优势

从阶段 1 的稳定分歧、方向翻转与部分翻转中固定选择 16 条任务，再做两轮反序 c6 复测：此前的站点级差异是否会在独立 holdout 中继续出现？

原始问题

Lexmount 的领先到底来自什么？

若只是随机波动，稳定方向应在新重复中快速消失；若是机制差异，同一站点应持续偏向同一后端。

短答案

方向差异大多复现，但原因是“双向站点效应”，不是 Lexmount 通用更强。

两轮 holdout 复现了 16/18 次预设稳定方向：Lexmount 侧 9/10，Local 侧 7/8。四个任务四轮均 Lexmount-only，三个任务四轮均 Local-only。前者的 Local 失败多伴随验证码、拒绝访问或网络导航证据；后者的 Lexmount 失败多伴随 Agent/Browser 控制终止，另有内容证据不足案例。

23 / 31

  • 验证码或机器人验证指标：8 次。
  • HTTP 拒绝访问指标：7 次。

11 / 22

11 次出现明确运行签名；其余 11 次没有运行签名，主要落在 mixed 或证据不足。

  • IGN 两条任务四轮 Local-only；豆瓣 task 3022 更像内容证据不足，而非固定控制终止。

稳定偏 Lexmount 的可重复信号

  • GameSpot：Local 四轮均出现 E1，且四轮均有验证码指标。
  • Vimeo：Local 四轮均出现 E3 与网络导航指标。
  • 3DM：Local 四轮均 E1，前三轮有 HTTP 拒绝指标。
  • 万方：四轮均 Lexmount-only，但没有一致原始日志指纹，根因置信度较低。

稳定偏 Local 的可重复信号

  • IGN task 125：Lexmount 三次未正常 done，一次导航失败。
  • IGN task 155：两次状态处理失败、两次未正常 done。
  • 豆瓣 task 3022：Agent 能运行，但 Judge 多次认为证据/内容不完整。

没有证据支持“Lexmount 约 300 秒固定 TTL”解释。

相关失败持续时间并不聚集在单一阈值，任务配置上限是 600 秒，provider 配置也没有设置 session TTL。当前可说的是 Browser/Agent 控制链出现了可重复终止签名，不能把它进一步归因为 Lexmount 平台 TTL。

05 / judge integrity

一条内容策略误拦截被隔离恢复，没有当作浏览器 0 分

Pair D Lexmount 的 task 52 首次形成 synthetic Judge failure；问题发生在评测传输层，而非 Agent 轨迹。

触发

GPT-5.4 多模态 Judge 请求被上游 Azure 内容策略判为 jailbreak，自动重试仍稳定失败。

隔离

保留 15 条有效 Judge 记录，不重跑 Agent，不改 task 52 的轨迹或最终答案。

恢复

同一 GPT-5.4、同一 stepwise rubric，显式 --api-max-images 0，仅使用非视觉运行证据与 Agent 最终答案补判 task 52。

结果

task 52 得分 64，阈值 60；Pair D Lexmount 从 synthetic 10/16 恢复为有效 11/16，synthetic failure remaining = 0。

这是评测完整性修复，不是“文字 Judge 等价于多模态 Judge”的证明。

干预只影响 64 次全四轮 Lexmount 尝试中的 1 次 Judge 记录，并在证据包中单独留账。后续应修复 evaluator 对 YAML api_max_images 的传参或通用内容策略 fallback。

06 / interpretation

最稳妥的结论是“互补优势”，不是单一浏览器排名

阶段 1 已经排除了总体巨大差值；阶段 2 进一步说明，总分抵消背后确实有可重复的站点和控制链差异。

证据支持

  • 一组站点上，Lexmount 持续降低 Local 的验证码、拒绝访问或导航失败风险。
  • 另一组任务上，Local 持续避开 Lexmount/Browser-use 控制链终止或证据不完整。
  • 两个 holdout 的反序执行仍复现 88.9% 的预设稳定方向。
  • c6 下 Lexmount session 生命周期干净，runner CPU/PSS 继续低于 Local。

证据不支持

  • 把 38/64 对 29/64 写成 LexBench 总体 +14.06pp。
  • 把访问指标直接归因于 IP、代理或某一浏览器实现。
  • 把控制终止直接归因于固定 session TTL。
  • 用 runner PSS 代表 Lexmount 端到端总资源。
当前产品表述

Lexmount 在部分公开站点上提供了可重复的访问稳定性收益，并把浏览器 CPU/内存从 runner 移出；Local Chrome 在少数任务上对当前 Agent/Browser 控制链更稳。两类效应并存，因此需要用站点路由、失败 fallback 和高并发数据来评价稳定性，而不是只看单次总成功率。
